1. API Overview
1.1 Introduction
The Indipips REST API enables all communication between the React frontend, mobile clients, and third-party integrations. Every feature of the Indipips platform — trader registration, challenge management, real-time risk monitoring, payments, payouts, KYC verification, and admin operations — is powered by this API. This document provides the complete reference for every endpoint including request formats, response schemas, authentication requirements, and error handling.

1.2 Base URLs

Same as production but with test data
Development
http://localhost:5000/api/v1
No TLS required in local dev

1.3 Authentication
All protected endpoints require a valid JWT access token in the Authorization header. Tokens are obtained via POST /auth/login or POST /auth/register. Tokens expire after 7 days — use the refresh endpoint to get a new token without re-login.

  REQUEST HEADER — AUTHENTICATION
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

1.4 Standard Response Envelope
Every API response — success or error — follows the same envelope structure. This allows the frontend to handle responses uniformly.

2.4 Refresh Access Token
POST
/auth/refresh
Auth: Public — requires valid refreshToken

Issues a new access token and refresh token using a valid, non-expired, non-blacklisted refresh token. Implements token rotation — the submitted refresh token is blacklisted and replaced. Call this when the access token expires or within 24 hours before expiry to maintain session.

  REQUEST BODY
{ "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." }

Success Response — 200 OK
  RESPONSE — 200 OK
{ "success": true, "message": "Token refreshed successfully.",
  "data": { "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
            "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "expiresIn": "7d" } }

401
Invalid, expired, or already-used (blacklisted) refresh token

2.5 Logout
POST
/auth/logout
Auth: Trader

Blacklists the current access token in Redis and clears the refresh token. Ensures the token cannot be reused even before natural expiry.

  REQUEST
// No request body — token from Authorization header is blacklisted

  RESPONSE — 200 OK
{ "success": true, "message": "Logged out successfully." }

10. Webhook Endpoints (System-to-System)
Webhook endpoints receive asynchronous events from Razorpay and broker systems. These are NOT called by frontend clients — they are called by external services. All webhooks verify cryptographic signatures before processing.

10.1 Razorpay Payment Webhook
POST
/webhooks/razorpay
Auth: No JWT — Razorpay Signature Verification

Receives payment and payout status events from Razorpay. Always returns 200 OK immediately after signature verification — processing happens async via Bull queue. Never return 4xx to Razorpay unless signature is invalid — this prevents retries for already-processed events.

Signature Verification
  SIGNATURE VERIFICATION CODE
const crypto = require('crypto');

function verifyRazorpaySignature(payload, signature) {
  const expectedSig = crypto
    .createHmac('sha256', process.env.RAZORPAY_WEBHOOK_SECRET)
    .update(JSON.stringify(payload))
    .digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(expectedSig), Buffer.from(signature)
  );
}

14. Integration Guide — Axios Setup
Recommended Axios configuration for the Indipips frontend. This setup handles authentication token injection, automatic refresh on 401, and consistent error handling.

  AXIOS INSTANCE — src/api/axios-instance.js
// src/api/axios-instance.js
import axios from 'axios';

const api = axios.create({
  baseURL: import.meta.env.VITE_API_URL,
  headers: { 'Content-Type': 'application/json' },
  timeout: 15000
});

// Request interceptor — inject access token
api.interceptors.request.use((config) => {
  const token = localStorage.getItem('accessToken');
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// Response interceptor — handle 401 with refresh
api.interceptors.response.use(
  (response) => response,
  async (error) => {
    const original = error.config;
    if (error.response?.status === 401 && !original._retry) {
      original._retry = true;
      try {
        const refreshToken = localStorage.getItem('refreshToken');
        const { data } = await axios.post(`${import.meta.env.VITE_API_URL}/auth/refresh`,
          { refreshToken });
        localStorage.setItem('accessToken', data.data.tokens.accessToken);
        localStorage.setItem('refreshToken', data.data.tokens.refreshToken);
        original.headers.Authorization = `Bearer ${data.data.tokens.accessToken}`;
        return api(original);
      } catch {
        localStorage.clear();
        window.location.href = '/login';
      }
    }
    return Promise.reject(error);
  }
);

export default api;

1. Database Overview
Indipips uses PostgreSQL 15 as its primary relational database, managed through Prisma ORM. The database is the single source of truth for all platform data. It is designed to handle high-volume financial transactions with strict ACID compliance, complete referential integrity, and query performance optimized for the real-time risk monitoring requirements.

1.1 Core Design Principles
All monetary values stored as BigInt in paise (100 paise = 1 Indian rupee) to completely eliminate floating-point precision errors that are catastrophic in financial calculations
UUID primary keys (v4) for all tables — prevents enumeration attacks and supports future distributed system architecture
Soft deletes only — isActive boolean flag — no hard deletes on any user or financial data
All tables include createdAt timestamp; all mutable tables include updatedAt with @updatedAt auto-management
Foreign key constraints enforced at database level — Prisma cannot be bypassed
Append-only financial records — Payment and Payout tables never have rows updated after creation
Sensitive columns (bank account, API keys) stored in encrypted form using application-level AES-256 before DB insert
All indexes documented and justified — no redundant indexes that slow write performance

6.1 Migration Approach
Prisma Migrate is used for all schema changes. Every migration creates a timestamped .sql file in the prisma/migrations directory. Migrations are version-controlled in Git alongside application code.

6.2 Safe Migration Rules
NEVER rename a column directly — add new column, migrate data, then drop old column
NEVER change column data type directly in production — add new column with new type, backfill, then swap
ALWAYS run migrations on staging environment and verify for at least 24 hours before production
ALWAYS take manual database backup before running production migrations
NEVER delete a column unless it has been marked as deprecated for at least 1 sprint
ALWAYS use --name flag with descriptive migration names — e.g., add_device_fingerprint_table
NEVER include data migrations (UPDATE/INSERT) in schema migration files — use separate seed scripts

1. Architecture Overview
The Indipips System Architecture is designed around four core principles: reliability for financial operations, real-time responsiveness for risk monitoring, security at every layer, and horizontal scalability from 100 to 100,000 traders without architectural changes. This document defines every component, service, integration, data flow, and infrastructure decision for the Indipips platform.

1.1 Architecture Style
Indipips uses a Modular Monolith architecture for the MVP phase — a single deployable Node.js application internally organized into clearly separated service modules. This provides the development simplicity of a monolith with the logical separation of microservices, enabling easy decomposition into true microservices post-investment when traffic demands it. The frontend is a completely separate React SPA.

1. Security Philosophy
Security at Indipips is not a checkbox — it is the foundation. Every feature, every API endpoint, every database column, and every third-party integration must be designed with security as the primary constraint, not an afterthought.

Indipips handles the most sensitive category of personal data in India — financial information, government identity documents, bank account details, and real monetary transactions. A single security breach could destroy trader trust, trigger regulatory action, and end the business. Therefore, Indipips implements defense-in-depth security with a minimum of 8 independent security layers, any one of which can prevent or detect an attack even if others fail.

1.1 Security Objectives
Protect trader financial data and identity documents from unauthorized access at all times
Ensure all monetary transactions (payments and payouts) are tamper-proof and auditable
Prevent unauthorized access to trader accounts through strong authentication and device verification
Detect and respond to security incidents within 4 hours of discovery
Maintain full compliance with all applicable Indian regulations (UIDAI, PMLA, PDPB, IT Act)
Achieve and maintain zero critical security vulnerabilities in the codebase at all times

4. Risk Engine — Complete Technical Specification
4.1 Risk Engine Architecture
The Risk Management Engine (RME) is the most critical and performance-sensitive component of Indipips. It must evaluate 8 trading rules in real-time within 5 seconds of every trade execution. The RME is implemented as a dedicated service module that is triggered via the webhook controller.

4.3 Risk Engine Code Flow
// risk-engine.service.js — Core evaluation function
async function evaluateTrade(challengeId, newTrade) {
  // Step 1: Load challenge from Redis cache (5-min TTL)
  let challenge = await redis.get('challenge:' + challengeId);
  if (!challenge) challenge = await prisma.challenge.findUnique({ where: { id: challengeId } });

  // Step 2: Calculate new values
  const newDailyPnl = challenge.dailyPnl + newTrade.pnl;
  const newTotalPnl = challenge.totalPnl + newTrade.pnl;
  const newBalance = challenge.accountSize + newTotalPnl;
  const newPeak = Math.max(challenge.peakBalance, newBalance);

  // Step 3: Check DAILY LOSS LIMIT (RME-001)
  if (newDailyPnl < -(challenge.accountSize * 0.04)) {
    await suspendChallenge(challengeId, 'DAILY_LOSS_LIMIT', 'temporary');
    await notifyTrader(challenge.userId, 'DAILY_LOSS_HIT');
    return { status: 'SUSPENDED', reason: 'Daily loss limit reached' };
  }

  // Step 4: Check MAX DRAWDOWN (RME-002)
  const drawdown = newPeak - newBalance;
  if (drawdown > challenge.accountSize * 0.08) {
    await failChallenge(challengeId, 'MAX_DRAWDOWN_EXCEEDED');
    await notifyTrader(challenge.userId, 'CHALLENGE_FAILED');
    return { status: 'FAILED', reason: 'Maximum drawdown exceeded' };
  }

  // Step 5: Update challenge in DB (atomic transaction)
  const updated = await prisma.challenge.update({
    where: { id: challengeId },
    data: { currentBalance: newBalance, peakBalance: newPeak,
            totalPnl: newTotalPnl, dailyPnl: newDailyPnl }
  });

  // Step 6: Update Redis cache
  await redis.setex('challenge:' + challengeId, 300, JSON.stringify(updated));

  // Step 7: Emit real-time update via Socket.io
  io.to('challenge:' + challengeId).emit('trade:update', {
    pnl: newTotalPnl / 100, dailyPnl: newDailyPnl / 100,
    balance: newBalance / 100, drawdown: drawdown / 100
  });

  // Step 8: Check PROFIT TARGET (RME-003) — non-violation
  if (newTotalPnl >= challenge.accountSize * 0.08 && challenge.daysTraded >= 5) {
    await evaluatePassConditions(challengeId);
  }
}
